# Staccato Constitution

## Core Principles

### I. Clean Project Boundaries (NON-NEGOTIABLE)

The solution MUST contain exactly 9 projects. Every project has a single,
exclusive role. Project references MUST follow the dependency map below with
zero exceptions — any violation corrupts the architectural contract and MUST
be corrected before merging.

Mandatory dependency rules:
- Application → Api, Domain, Repository, Persistence
- Api          → Domain, ApiModels, DomainModels
- Domain       → DomainModels ONLY
- Repository   → Domain, EntityModels, Persistence
- Persistence  → EntityModels
- ApiModels    → (no project references)
- DomainModels → (no project references)
- EntityModels → (no project references)
- Tests        → all projects

Domain MUST NOT reference Api, ApiModels, Repository, Persistence, or
EntityModels under any circumstance. Introducing such a reference inverts
the dependency flow and makes Domain untestable in isolation.

**Rationale**: Enforcing strict project boundaries prevents circular
dependencies, keeps the domain layer framework-agnostic, and ensures that
every layer can be tested without materialising its dependants.

### II. Layered Architecture — Service / Repository / Unit of Work (NON-NEGOTIABLE)

Every domain operation MUST be exposed through a service interface defined
in `Domain/Services/`. Service implementations MUST live alongside their
interfaces in the same folder. Controllers MUST call service interfaces only
— they MUST NOT call repositories directly and MUST NOT contain business logic.

Repository interfaces MUST be defined in `Domain/Interfaces/Repositories/`.
Concrete implementations MUST live in `Repository/Repositories/`. Repository
methods MUST NOT call `SaveChanges` directly.

`IUnitOfWork` MUST be defined in `Domain/Interfaces/` with a single method:
`Task<int> CommitAsync(CancellationToken ct)`. Services MUST call
`IUnitOfWork.CommitAsync` to persist changes.

**Rationale**: The three-pattern stack (Service / Repository / UoW) keeps
persistence concerns out of business logic, makes services independently
unit-testable via Moq, and gives a single, explicit commit point per operation.

### III. Domain Purity

Domain is the clean core of the application. It MUST contain:
service interfaces, service implementations, repository interfaces,
`IUnitOfWork`, domain exceptions (inheriting `BusinessException`), and all
business rule logic.

Domain MUST NOT contain: controllers, DTOs, EF Core types, HTTP status codes,
`IActionResult`, `HttpContext`, or any infrastructure concern. Services MUST
throw typed exceptions (subclasses of `BusinessException`) — they MUST NOT
return `IActionResult` or set HTTP status codes.

All business rule enforcement (grid validation, building-block constraints,
module-type invariants, ownership checks) MUST live in the service layer.

**Rationale**: A pure domain makes business logic portable, testable without
the web stack, and readable without infrastructure noise.

### IV. Data Model Integrity

The entity hierarchy is immutable in structure:
`User → Notebook → Lesson → LessonPage → Module → ContentJson`.

Every primary key MUST be a `Guid` generated by the application
(`Guid.NewGuid()`), never by the database. All datetimes MUST be stored
and returned as UTC (`DateTime.UtcNow`).

JSON columns (`ContentJson`, `PositionsJson`, `StylesJson`, `LessonIdsJson`)
MUST be stored as `NVARCHAR(MAX)`, serialised with `System.Text.Json`, and
configured via `.HasColumnType("nvarchar(max)")`.

Cascade delete rules MUST be enforced in EF Core configuration:
- Notebook deleted → Lessons, LessonPages, Modules, NotebookModuleStyles deleted
- Lesson deleted → LessonPages and Modules deleted
- LessonPage deleted → Modules deleted
- User deleted → all Notebooks (and cascade) deleted
- Chord/Instrument deleted → RESTRICT (these entities are immutable post-seed)

Immutable fields (MUST NOT be changeable after creation):
`Notebook.PageSize`, `Notebook.InstrumentId`, the entire `Chord` entity,
the entire `Instrument` entity.

Every `Notebook` MUST have exactly 12 `NotebookModuleStyle` records
(one per `ModuleType` enum value) created atomically at notebook creation.

Every `Lesson` MUST have at least one `LessonPage`. `LessonPage` with
`PageNumber = 1` MUST be created automatically when a `Lesson` is created.

**Rationale**: Explicit invariants prevent data corruption that would be
expensive to detect and repair at runtime or in production.

### V. API Contract Discipline

HTTP status codes MUST follow these exact conventions:
- `200` — successful GET or PUT
- `201` — successful POST that creates a resource
- `202` — accepted async operation (PDF export)
- `204` — successful DELETE or action with no return value

Successful responses MUST return the resource or array directly (no envelope
wrapper). The only exception is `LessonPage` creation, which MUST use
`{ data: T, warning: string | null }` when the 10-page soft limit applies.

Business rule errors MUST use the format:
`{ "code": "ERROR_CODE", "message": "Localized message.", "details": {} }`.
`code` MUST always be English. `message` MUST be localised.

Infrastructure/unexpected errors MUST use RFC 7807 Problem Details.

FluentValidation errors MUST use the standard ASP.NET model-state format:
`{ "errors": { "fieldName": ["message"] } }`.

All protected endpoints MUST use `[Authorize]`. User identity MUST be
extracted from JWT claims via `User.FindFirstValue(ClaimTypes.NameIdentifier)`.
Ownership violations MUST return `403` (never `404`).

Rate limiting MUST apply to `/auth/*` endpoints only: 10 requests per
minute per IP using ASP.NET Core's built-in middleware.

CORS allowed origins MUST be loaded from `CorsOptions.AllowedOrigins`
in `appsettings.json`.

**Rationale**: Consistent, predictable HTTP contracts reduce frontend
integration bugs and make the API self-documenting through its status codes.

### VI. Content Model Strictness

Module content MUST be stored as a JSON array of typed building-block
objects in `Module.ContentJson`. Each object MUST include a `type`
discriminator string matching the `BuildingBlockType` enum.

ALL user-entered text throughout the application is plain text with optional
bold only. No italic, underline, colour, font size, or any other formatting
is permitted. Text MUST be stored as `TextSpan[]`:
`{ "text": string, "bold": boolean }`.

The static class `ModuleTypeConstraints` in `DomainModels/Constants/` MUST
be the single source of truth for:
- Which `BuildingBlockType` values are allowed per `ModuleType`
- Minimum grid dimensions (`MinWidth`, `MinHeight`) per `ModuleType`

The static class `PageSizeDimensions` in `DomainModels/Constants/` MUST be
the single source of truth for page grid dimensions per `PageSize` enum value.

The grid uses a fixed 5mm dot spacing (not configurable). All module
positions and dimensions MUST be stored as integer grid-unit values.

Module placement MUST be validated server-side in `ModuleService`:
1. `gridWidth >= ModuleTypeConstraints.MinimumSizes[type].MinWidth`
2. `gridHeight >= ModuleTypeConstraints.MinimumSizes[type].MinHeight`
3. `gridX >= 0` AND `gridY >= 0`
4. `gridX + gridWidth <= pageGridWidth`
5. `gridY + gridHeight <= pageGridHeight`
6. No overlap (checked via `IModuleRepository.CheckOverlapAsync`)

The `Breadcrumb` module's `ContentJson` MUST always be an empty array `[]`.
A lesson MUST contain at most one `Title` module across all its pages.

**Rationale**: Strict content constraints guarantee that every module can
be rendered by the frontend without defensive branching, and prevent
data-model divergence as new features are added.

### VII. Security by Design

Access tokens MUST live in JavaScript memory only — never in `localStorage`
or `sessionStorage`. Refresh tokens MUST be stored exclusively in the
`staccato_refresh` HttpOnly cookie (`SameSite=Strict`, `Secure=true` in
production). Every call to `POST /auth/refresh` MUST revoke the old refresh
token and issue a new one atomically (rotation).

All secrets (JWT signing key, Azure Blob connection string, Google Client ID)
MUST live in `appsettings.json` and MUST be injected via `IOptions<T>`.
Hardcoded secrets in source code are prohibited.

Passwords MUST be hashed with BCrypt at work factor 12.

Google OAuth MUST be validated server-side with
`GoogleJsonWebSignature.ValidateAsync`. The frontend's decoded token claims
MUST NOT be trusted without server-side validation.

User accounts use soft delete: `DELETE /users/me` sets
`ScheduledDeletionAt = UtcNow + 30 days`. `AccountDeletionCleanupService`
runs daily and hard-deletes accounts past their deadline, including
Azure Blob avatar deletion. All other entities use hard delete (cascade).

Chord and Instrument entities MUST have `DeleteBehavior.Restrict` — they
are immutable post-seed and must never be deleted.

**Rationale**: Explicit security rules prevent the most common classes of
token theft, credential exposure, and data loss in a multi-user application.

### VIII. Test Discipline

Unit tests (in `Tests/Unit/`) MUST test service-layer logic in isolation
using Moq for all dependencies. Every service MUST have a corresponding
test class. Parameterised scenarios MUST use `[Theory]` + `[InlineData]`.
Both the happy path AND all exception/error paths MUST be covered.

Integration tests (in `Tests/Integration/`) MUST use a custom
`WebApplicationFactory<Program>` with the InMemory EF Core provider.
Every controller MUST have a corresponding integration test class.
All authenticated endpoint tests MUST inject a test JWT via a helper.
Tests MUST be fully independent — unique `Guid`s for all seeded test data.
Key business rules (overlap rejection, minimum size, duplicate Title module,
boundary check) MUST be tested end-to-end.

**Rationale**: Two-tier testing ensures both the correctness of business
logic in isolation and the correctness of the full HTTP contract under
realistic conditions.

### IX. Code Quality Standards

- Nullable reference types MUST be enabled in every project:
  `<Nullable>enable</Nullable>`.
- File-scoped namespaces MUST be used throughout.
- Primary constructors SHOULD be used where they reduce boilerplate.
- `async`/`await` MUST be used consistently. `.Result` and `.Wait()` are
  prohibited.
- Every public method in a service or repository MUST accept a
  `CancellationToken ct = default` parameter and pass it through to all
  EF Core and async I/O calls.
- Magic strings for error codes, module type names, or policy names are
  prohibited — use constants or enums.
- Controllers MUST be thin: extract input → validate (FluentValidation
  pipeline) → call service → map response → return. Nothing else.
- `HttpContext` MUST NOT be used outside controllers and middleware.
- Configuration MUST be injected via `IOptions<T>`. Direct use of
  `IConfiguration` in services or repositories is prohibited.
- No new major library or framework may be introduced without updating
  this constitution.

**Rationale**: Consistent code quality rules reduce cognitive load when
switching between layers, prevent common async bugs, and keep the codebase
maintainable as it grows.

## Technology Stack

**Runtime**: .NET 10, ASP.NET Core 10
**Language**: C# 13 with nullable reference types enabled throughout
**Database**: SQL Server via Entity Framework Core 10
**ORM mapping**: AutoMapper
- `EntityModel → DomainModel` in the Repository layer
- `DomainModel → Response DTO` in the Api layer
**Validation**: FluentValidation — all validators in ApiModels; auto-validation
pipeline configured in Application
**Authentication**: JWT Bearer HS256 (15-min access token) + HttpOnly refresh
cookie `staccato_refresh` (7-day sliding; 30-day with rememberMe)
**Google OAuth**: `Google.Apis.Auth` — server-side `GoogleJsonWebSignature.ValidateAsync`
**Password hashing**: `BCrypt.Net-Next` at work factor 12
**PDF generation**: QuestPDF — synchronous rendering inside async background job
**File storage**: Azure Blob Storage via `Azure.Storage.Blobs`
**Real-time**: ASP.NET Core SignalR
**Background jobs**: `IHostedService` + `Channel<T>` (in-process; Hangfire prohibited)
**Error format (infrastructure)**: RFC 7807 Problem Details
**Error format (business)**: `{ code, message, details }` custom envelope
**Localization**: `IStringLocalizer` + `.resx` resource files for `en` and `hu`;
language resolved from `Accept-Language` request header
**Testing**: xUnit + Moq + `Microsoft.AspNetCore.Mvc.Testing` +
`Microsoft.EntityFrameworkCore.InMemory`

No library outside this list may be introduced without a constitution amendment.

## Naming Conventions

**Classes and files:**

| Artefact | Convention | Example |
|---|---|---|
| EF Core entity | suffix `Entity` | `NotebookEntity` |
| Domain model | no suffix | `Notebook` |
| Repository interface | prefix `I`, suffix `Repository` | `INotebookRepository` |
| Repository implementation | suffix `Repository` | `NotebookRepository` |
| Service interface | prefix `I`, suffix `Service` | `INotebookService` |
| Service implementation | suffix `Service` | `NotebookService` |
| Request DTO | suffix `Request` | `CreateNotebookRequest` |
| Response DTO | suffix `Response` | `NotebookDetailResponse` |
| FluentValidation validator | suffix `Validator` | `CreateNotebookRequestValidator` |
| EF Core configuration | suffix `Configuration` | `NotebookConfiguration` |
| AutoMapper profile | suffix `Profile` | `EntityToDomainProfile` |
| Background service | suffix `Service` | `PdfExportBackgroundService` |
| Exception | suffix `Exception` | `NotFoundException` |

Namespaces MUST follow folder structure: `ProjectName.FolderName.SubfolderName`.

**Database:**
- Table names: plural PascalCase, no `Entity` suffix (e.g. `Notebooks`, `Lessons`)
- Column names: PascalCase matching the property name
- Primary keys: always `Guid`, always named `Id`
- Foreign keys: `{EntityName}Id` (e.g. `NotebookId`, `UserId`)

**API routes:**
- Lowercase, hyphen-separated multi-word segments (e.g. `/lesson-pages`)
- No trailing slashes
- No version prefix (versioning deferred)
- Collections: plural nouns (e.g. `/notebooks`)
- Single resources: include the ID (e.g. `/notebooks/{id}`)
- Sub-resources: nest under parent (e.g. `/notebooks/{id}/lessons`)
- Actions (non-REST): verb-noun kebab-case (e.g. `/cancel-deletion`)

## Governance

This constitution supersedes all other project conventions, README guidance,
and prior verbal agreements. In cases of conflict, this document prevails.

**Amendment procedure:**
1. Propose the change in a pull request that modifies this file.
2. State the version bump type (MAJOR / MINOR / PATCH) and rationale.
3. Run the consistency propagation checklist (see below) and record results
   in the Sync Impact Report comment at the top of this file.
4. Update `LAST_AMENDED_DATE` and `CONSTITUTION_VERSION`.
5. Obtain review approval before merging.

**Versioning policy:**
- MAJOR: backward-incompatible governance change — principle removal,
  redefinition, or dependency-rule change.
- MINOR: new principle or section added, or material expansion of existing
  guidance.
- PATCH: clarification, wording improvement, or typo fix.

**Consistency propagation checklist (run on every amendment):**
- [ ] `.specify/templates/plan-template.md` — Constitution Check gates reflect
      current principles
- [ ] `.specify/templates/spec-template.md` — mandatory sections match current
      constraints
- [ ] `.specify/templates/tasks-template.md` — task categories reflect current
      principle-driven requirements
- [ ] `STACCATO_FRONTEND_DOCUMENTATION.md` — no contradiction with amended rules

**Compliance review:**
Every pull request MUST pass the Constitution Check gates defined in
`.specify/templates/plan-template.md` before merge. Any deliberate deviation
MUST be recorded in the plan's Complexity Tracking table with justification.

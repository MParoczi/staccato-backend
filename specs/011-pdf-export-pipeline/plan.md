# Implementation Plan: PDF Export Pipeline

**Branch**: `011-pdf-export-pipeline` | **Date**: 2026-03-29 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/011-pdf-export-pipeline/spec.md`

## Summary

Implement the asynchronous PDF export pipeline: users queue an export via `POST /exports`, a background service renders the notebook as a PDF using QuestPDF, uploads it to Azure Blob Storage, and notifies the user via SignalR. Exports expire after 24 hours and are cleaned up by a daily background service. The PDF faithfully renders the notebook's cover page, index, and lesson pages with modules at exact grid positions, including all 10 building block types.

## Technical Context

**Language/Version**: C# 13 / .NET 10, ASP.NET Core 10
**Primary Dependencies**: QuestPDF (PDF rendering), Azure.Storage.Blobs (file storage), ASP.NET Core SignalR (real-time notifications), System.Threading.Channels (job queue)
**Storage**: SQL Server via EF Core 10 (existing `PdfExports` table), Azure Blob Storage (PDF files)
**Testing**: xUnit + Moq + WebApplicationFactory + InMemory EF Core
**Target Platform**: Windows/Linux server
**Project Type**: Web service (ASP.NET Core WebAPI)
**Performance Goals**: Export ≤30 seconds for notebooks with up to 50 pages
**Constraints**: Single-process in-memory queue (no distributed queue); 24-hour PDF expiry; one active export per notebook
**Scale/Scope**: Single-user exports, low concurrency (bounded channel capacity 50)

**Structure Decision**: Follows the existing 9-project architecture. All new code fits into existing projects. PDF rendering classes are grouped under `Application/Pdf/` since they depend on QuestPDF (infrastructure). The `IPdfExportQueue` interface in Domain abstracts the channel to maintain Domain purity.

### Key Design Decisions

1. **Channel abstraction**: `IPdfExportQueue` in Domain → `PdfExportChannel` in Application. Domain services enqueue via interface; background service reads directly from channel. Registered as singleton with forwarded interface.

2. **PDF rendering in Application**: QuestPDF renderers live in `Application/Pdf/` because they are infrastructure (PDF library). They consume DomainModels read-only. No business logic in renderers.

3. **PdfDataLoader in Application**: Aggregates data from multiple repositories into a single `PdfExportData` record for rendering. This is query orchestration, not business logic, so it lives in Application alongside the renderers that consume it.

4. **Background service scoping**: `PdfExportBackgroundService` creates `IServiceScope` per job to resolve scoped dependencies (repositories, DbContext, UnitOfWork). The channel and SignalR hub context are singletons injected directly.

5. **Stale recovery on startup**: Before entering the channel reader loop, the background service queries for Processing exports and resets them to Pending + re-enqueues them.

6. **Grid-to-mm conversion**: PDF rendering converts grid units to millimeters using `PageSizeDimensions` — `widthMm / gridWidth * gridUnits` gives the mm position. QuestPDF natively supports mm units.

7. **Notebook title as filename**: Downloaded PDF uses `$"{notebook.Title}.pdf"` with filename-unsafe characters replaced. Loaded from notebook on download.

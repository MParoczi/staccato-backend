- .NET 10 SDK installed

## Local Development Setup

### 1. Verify SQL Server Connection String

Check `Application/appsettings.Development.json` (or `appsettings.json`):
```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=(localdb)\\mssqllocaldb;Database=Staccato;Trusted_Connection=True;"
  }
}
```

Adjust if using SQL Server Docker:
```json
"DefaultConnection": "Server=localhost,1433;Database=Staccato;User Id=sa;Password=YourPassword;TrustServerCertificate=True;"
```

### 2. Build the Solution

```bash
dotnet build Staccato.sln
```

### 3. Run the Application (Database Auto-Initializes)

```bash
dotnet run --project Application/Application.csproj
```

On first run, `DbInitializer` executes before the app starts accepting requests:
1. Creates the database if it does not exist
2. Applies the `InitialCreate` migration
3. Seeds 7 instruments
4. Seeds guitar chords from `guitar_chords.json`
5. Seeds 5 system style presets

### 4. Verify Seed Data (Optional)

Connect to the database with SQL Server Management Studio or `sqlcmd`:
```sql
SELECT COUNT(*) FROM Instruments;          -- Expected: 7
SELECT COUNT(*) FROM Chords;               -- Expected: ~200+
SELECT COUNT(*) FROM SystemStylePresets;   -- Expected: 5
SELECT Name, IsDefault FROM SystemStylePresets ORDER BY DisplayOrder;
```

## Adding a New Migration (Future Development)

After changing entity configurations:

```bash
dotnet ef migrations add <MigrationName> \
  --project Persistence/Persistence.csproj \
  --startup-project Application/Application.csproj
```

Apply pending migrations:
```bash
dotnet ef database update \
  --project Persistence/Persistence.csproj \
  --startup-project Application/Application.csproj
```

## Running Tests

```bash
# All tests
dotnet test Staccato.sln

# Unit tests only
dotnet test Tests/Tests.csproj --filter "FullyQualifiedName~Unit"

# Integration tests (use InMemory EF — no SQL Server required)
dotnet test Tests/Tests.csproj --filter "FullyQualifiedName~Integration"
```

Integration tests use `Microsoft.EntityFrameworkCore.InMemory` — no SQL Server needed for test execution. The `guitar_chords.json` file must be present (copied to output directory) for `ChordSeeder` tests.

## Key Implementation Notes for Developers

**JSON columns**: All `*Json` properties are `string` or `string?` at the entity level. Serialization/deserialization happens in the repository layer using `System.Text.Json`. Do not configure EF Core owned types or JSON document mappings — raw `nvarchar(max)` strings only.

**Enum storage**: All enums stored as integers (EF Core default). Do not add `.HasConversion<string>()` — this would break the `HasFilter` expressions on `PdfExportEntity` that use integer comparisons.

**Filtered indexes**: `HasFilter` expressions use SQL column names (bracket-quoted), not C# property names. Verify generated migration SQL matches expected `WHERE` clause.

**Seeder idempotency**: Each seeder guards with a `COUNT(*)` check — runs only if the table is empty. If partial data exists (e.g., 3 instruments seeded), the seeder skips entirely. This is intentional: assume partial seed is a data corruption issue, not a partial run.

**DbInitializer scope**: `DbInitializer` is registered as a scoped service. The startup invocation in `Program.cs` creates its own scope to avoid lifetime conflicts.

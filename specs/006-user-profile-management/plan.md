# Implementation Plan: User Profile Management

**Branch**: `006-user-profile-management` | **Date**: 2026-03-28 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/006-user-profile-management/spec.md`

## Summary

Add user profile management to the Staccato API: profile GET/PUT, account soft-delete with 30-day grace and cancellation, avatar upload/remove via Azure Blob (public URLs), user-saved style presets with per-user name uniqueness, and a daily `AccountDeletionCleanupService`. Requires two new nullable columns on `Users` (one migration), a new `IAzureBlobService` abstraction with implementation in `Application/Services/`, a new `IUserService`/`UserService`, a thin `UsersController`, AutoMapper response profiles, and FluentValidation request models. No new library dependencies.

## Technical Context

**Language/Version**: C# 13 / .NET 10
**Primary Dependencies**: ASP.NET Core 10, EF Core 10, AutoMapper, FluentValidation, Azure.Storage.Blobs (already in Application.csproj), xUnit + Moq

**Target Platform**: Linux/Windows server (ASP.NET Core 10 hosted)

**Performance Goals**: Profile GET/PUT < 2 s p95; avatar upload < 5 s for 2 MB under normal conditions
**Constraints**: 9-project limit preserved; no new packages; `Domain` references `DomainModels` only
**Scale/Scope**: Single in-process instance; no distributed locking needed

## Spec Alignment Decisions

| Decision | Rationale |
|---|---|
| `displayName` in spec → `firstName` + `lastName` in API | Entity established in feature 005 stores two fields; `AuthService.RegisterAsync` already splits displayName on first space. Exposing separate fields is more precise and avoids lossy round-trips. |
| `preferredLanguage` in spec → `language` (Language enum) in API | Entity field is typed `Language` enum (existing `en`/`hu` values). String-based "preferredLanguage" would require extra parsing. |
| `AzureBlobService` in `Application/Services/`, not `Persistence/Services/` | `Azure.Storage.Blobs` and `BlobServiceClient` are already wired in Application. Persistence only depends on EntityModels; adding Azure SDK there would pollute that layer. |
| ACCOUNT_DELETION_ALREADY_SCHEDULED → `ConflictException` (409) | Scheduling deletion when already scheduled is a state conflict — 409 is semantically correct. The spec's stated 400 is overridden per HTTP semantics; documented here as a deliberate deviation. |
| ACCOUNT_DELETION_NOT_SCHEDULED → `BadRequestException` (400) | Cancelling deletion when none is scheduled is invalid client state. A new `BadRequestException` (400) is required since no existing 400-status exception exists. |

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| `BadRequestException` (400) is a new exception type | ACCOUNT_DELETION_NOT_SCHEDULED must return 400 per spec. No existing 400 exception exists. | Using `ValidationException` (422) or `ConflictException` (409) would return the wrong status code for a client state error. |
| ACCOUNT_DELETION_ALREADY_SCHEDULED returns 409, not 400 | 409 Conflict is semantically correct for a state conflict. No existing 400 exception existed before this feature. | Would require creating BadRequestException first; 409 is more HTTP-correct and less likely to mislead API clients. |

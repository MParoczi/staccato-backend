## Summary

Populate the `DomainModels` project (currently empty) with all application-wide
enums, domain model classes, building block content types, and the two static
constant classes (`ModuleTypeConstraints`, `PageSizeDimensions`). No project
references are added — DomainModels must remain a zero-dependency class library.
The deliverable is a project that compiles clean with nullable reference types
enabled and is ready to be consumed by Domain, Repository, Api, EntityModels,
ApiModels, and Tests.

## Phase 0: Research Summary

> See [research.md](research.md) for full decision rationales.

Key decisions resolved:

1. **Allowed blocks per ModuleType** — authoritative mapping sourced from
   `STACCATO_FRONTEND_DOCUMENTATION.md §5.4`. The original research-derived
   four-group model (Structural / Callout / Content / Chord) was a best-effort
   inference made without the frontend docs and has been fully superseded.
   See `data-model.md → ModuleTypeConstraints.AllowedBlocks` and `research.md
   Decision 1` for the complete per-type table. Key rules: `Breadcrumb` = empty set;
   `ChordTablatureGroup` permitted only in ChordTablature, Practice, FreeText;
   `ChordProgression` permitted only in Practice, Example, FreeText; `FreeText`
   accepts all 10 BuildingBlockType values.

2. **CheckboxListItem** — standalone class with `List<TextSpan> Spans` and
   `bool IsChecked`; clarification Q1 confirmed persisted checked state.

3. **ChordProgressionBlock sub-types** — four standalone top-level classes:
   `ChordProgressionBlock`, `ChordProgressionSection`, `ChordMeasure`, `ChordBeat`.
   Clarification Q2 confirmed standalone (not nested inner classes).

4. **TableColumn** — standalone support class for `TableBlock`; rows stay as
   `List<List<List<TextSpan>>>` (no named Row/Cell wrappers — single-property
   wrappers add indirection without value).

5. **Module timestamps** — no `CreatedAt` or `UpdatedAt`; clarification Q3 confirmed.
   Change tracking at the Lesson level is sufficient.

6. **BuildingBlock.Type** — get-only abstract property set in each concrete
   constructor; no serialization attributes on domain models (FR-034).

## Phase 1: Design Summary

All types, properties, and constraint values are fully specified in [data-model.md](data-model.md).
Public type surface is documented in [contracts/type-hierarchy.md](contracts/type-hierarchy.md).

### Key Design Points

- `BuildingBlock` is `abstract` — cannot be instantiated directly.
- `TextSpan` is a `record` — value semantics per clarification Q4 (spec.md Clarifications, 2026-03-02). The earlier "plain class" note in this file was written before the clarification and has been superseded by spec.md FR-022 and tasks.md T025.
- `CheckboxListItem` and `ChordTablatureItem` are plain classes with init-only
  or writable properties — matching the style of all other domain models.
- `ModuleTypeConstraints` and `PageSizeDimensions` use `static readonly` fields
  initialized inline (no lazy init, no computed factory) — values are
  compile-time constants.
- `IReadOnlySet<BuildingBlockType>` is used for the allowed-blocks value type
  to prevent consumers from accidentally mutating the constraint table.

### Re-checked Constitution Gates Post-Design

All gates remain ✅. No design decision introduced any violation.

## Implementation Notes for `/speckit.tasks`

The following ordering must be respected in tasks:

1. **Enums first** — all building block and model files depend on enums.
2. **TextSpan and BuildingBlock base** — concrete building block types depend on both.
3. **Support types** — `CheckboxListItem`, `TableColumn`, `ChordBeat`, `ChordMeasure`,
   `ChordProgressionSection`, `ChordTablatureItem` before their parent blocks.
4. **Concrete building block classes** — after their support types.
5. **Domain model classes** — independent of building blocks (models only store
   `ContentJson` as a raw string).
6. **Constants** — after enums (depend on both enum types).
7. **Unit tests** — after constants (test coverage of constraint completeness).
8. **Build verification** — after all files written.

No parallel blockers within DomainModels itself (files are independent). The
unit tests cannot start until constants files are complete.

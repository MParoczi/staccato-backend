# Implementation Plan: Repository Pattern and Unit of Work

**Branch**: `004-repository-uow` | **Date**: 2026-03-08 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/004-repository-uow/spec.md`

---

## Summary

Introduce 11 typed repository interfaces and `IUnitOfWork` in the `Domain` project, with concrete EF Core implementations in the `Repository` project. A generic `RepositoryBase<TEntity, TDomain>` abstract class reduces boilerplate. A single `EntityToDomainProfile` with `ReverseMap()` handles both read-direction (Entity ‚Üí Domain) and write-direction (Domain ‚Üí Entity) AutoMapper mappings. All repositories are registered as Scoped in DI. Integration tests validate the two highest-risk methods (`CheckOverlapAsync` and `GetExpiredExportsAsync`).

---

## Technical Context

**Language/Version**: C# 13, .NET 10
**Primary Dependencies**: Entity Framework Core 10, AutoMapper, xUnit, Moq
**Storage**: SQL Server (existing `AppDbContext`; all 12 `DbSet<T>` properties already present)
**Testing**: xUnit + in-memory EF Core provider for repository integration tests
**Target Platform**: ASP.NET Core 10 hosted service (Linux/Windows server)
**Project Type**: Internal infrastructure layer (no new HTTP endpoints)
**Performance Goals**: All repository reads complete in < 100 ms under typical load; `CheckOverlapAsync` uses a server-side predicate (no client-side filtering)
**Constraints**: Domain project MUST NOT reference EntityModels, Repository, or Persistence. `RevokeAllForUserAsync` is the only repository method permitted to commit directly (via `ExecuteUpdateAsync`)
**Scale/Scope**: 11 repository interfaces + 11 implementations + 1 UoW + 1 AutoMapper profile + 4 exception subclasses + DI registration + 2 integration test classes

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| `RevokeAllForUserAsync` commits immediately via `ExecuteUpdateAsync` (bypasses UoW) | Bulk-revoking N tokens with change tracking is O(N) writes; logout-all-devices and account deletion need a single SQL UPDATE for correctness and performance | Using change tracking + UoW.CommitAsync would require the caller to not call CommitAsync again for this operation (confusing contract) and would be inefficient for users with many sessions |

## Phase 1: Design

### Key Design Points

**RepositoryBase pattern**: Each concrete repository calls `base(context, mapper)`. The base class implements the 4 generic methods using `context.Set<TEntity>()`. Subclasses add only their domain-specific query methods.

**Overlap detection predicate**:
```
existing.GridX < gridX + gridWidth
&& existing.GridX + existing.GridWidth > gridX
&& existing.GridY < gridY + gridHeight
&& existing.GridY + existing.GridHeight > gridY
```
This runs as a server-side SQL predicate via EF Core LINQ translation. No client-side filtering.

**Active export query**: `Status != ExportStatus.Failed` ‚Äî catches Pending, Processing, and Ready in one negative condition, which is forward-compatible if new statuses are added.

**Expired export query**: `Status != ExportStatus.Failed && CreatedAt < utcCutoff` ‚Äî deliberately excludes Failed exports from cleanup consideration (they may be referenced for error reporting).

**GetWithActiveTokensAsync active token filter**: `!IsRevoked && ExpiresAt > DateTime.UtcNow` ‚Äî this `DateTime.UtcNow` call is acceptable inside a read query (not a deterministic business rule; it represents "now" at query time).

### AutoMapper Profile Notes

- `PdfExport.LessonIds` (domain: `List<Guid>?`) ‚Üî `PdfExportEntity.LessonIdsJson` (entity: `string?`) ‚Äî these require custom value converters or `Ignore()` in the profile, with manual JSON serialization in the repository implementation.
- All other entity ‚Üî domain property names match exactly; `ReverseMap()` handles them automatically.
- Navigation properties on EntityModels (`Notebook`, `User`, etc.) are ignored by AutoMapper when mapping Domain ‚Üí Entity (domain models have no navigation properties to fill them).

### Exception Subclass Pattern

All exception subclasses follow the same shape:

```csharp
// BusinessException.StatusCode is `protected init` ‚Äî set in constructor body, not override
public class NotFoundException : BusinessException
{
    public NotFoundException(string message, object? details = null)
        : base("NOT_FOUND", message, details)
    {
        StatusCode = 404;
    }

    public NotFoundException() : this("Resource not found.") { }
}
```

Error codes for each subclass match the API Contract Discipline error codes defined in CLAUDE.md.
